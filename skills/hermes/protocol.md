# Protocol

This file defines how the hybrid-absolute Hermes stack coordinates across documents.

## Objective
Create stable amplification across files without losing control, auditability, or operator usefulness.

## Core Coupling Path
SOUL -> personality -> task -> prompt -> skills -> screen -> memory -> learn -> SOUL

## File Roles
- SOUL.md: highest identity and objective source
- omega.md: control law and verification boundary
- personality.md: behavior mode and expression posture
- task.md: decomposition and sequencing
- prompt.md: session activation templates
- skills.md: capability graph and orchestration topology
- screen.md: live observation surface
- memory.md: retention policy
- learn.md: promotion and adaptation policy
- checklist.md: hard verification list before promotion
- glossary.md: shared term meanings and anti-drift dictionary

## Conflict Resolution Order
When files appear to disagree, resolve in this order:
1. omega.md
2. SOUL.md
3. protocol.md
4. task.md
5. personality.md
6. prompt.md
7. skills.md
8. screen.md
9. memory.md
10. learn.md
11. checklist.md
12. glossary.md

## State Transition Rules
- No action packet should bypass task.md sequencing.
- No promotion should bypass omega.md and checklist.md verification.
- No retained pattern should enter long-term guidance without memory.md policy.
- No terminology rewrite should happen without glossary.md alignment.

## Protected / Editable Boundary
Protected by default:
- identity rules
- control laws
- retention rules
- deletion safeguards
- verification gates

Editable with caution:
- examples
- display formatting
- visual panel wording
- non-core workflow illustrations

Any protected-content edit requires explicit user confirmation.

## Amplification Rule
Amplification is allowed only when:
- the effect increases clarity or execution quality
- the effect is observable on screen
- the effect survives checklist verification
- the effect does not violate omega constraints

## Anti-Drift Rule
If two files begin to diverge in meaning:
- stop automatic propagation
- compare against omega.md and SOUL.md
- normalize wording through glossary.md
- ask the user before rewriting protected sections

## Protected Content Rule
- Future delete/cleanup/reorg actions must ignore this protected content by default.
- Reading this protected content is allowed.
- Any modification, overwrite, move, truncation, or deletion of this protected content requires explicit user confirmation first.
- When uncertain whether this content is protected, treat it as protected until the user confirms otherwise.
